# 红包/卡券功能设计方案（珍果链）

> 状态：已评审通过，进入实现阶段
> 适用范围：uni-app 小程序前端 + FastAPI 后端（`backend/`）

## 一、背景与目标

`pages/mine/index.vue` 的"我的"页面有一个纯静态的 **"0张 红包/卡券"** 占位卡片，背后没有任何逻辑，整个仓库此前没有任何卡券相关代码。

业务目标：

- **认证通过后自动给客户发券**；
- 客户在**订单结算时可以选择是否使用券、看到抵扣多少、下单时核销**；
- 提供**"我的卡券"**页面查看券；
- 提供**后台"券种管理"**页配置券模板。

### 本期范围（已确认）

| 决策项 | 结论 |
|---|---|
| 券类型 | **仅金额型**：满减 + 无门槛立减（门槛设 0 即无门槛）。不做百分比折扣券。 |
| 有效期 | **领取后 N 天**：发放时按模板 `valid_days` 自动计算到期日。 |
| 落地范围 | **完整打通**：认证送券 → 结算选券抵扣 → 下单核销 → 我的卡券，**并含后台券种管理页**。 |

### 关键技术约束

订单金额的**权威计算在服务端** `_apply_order_payload`（`backend/app/api/public.py`）。下单请求只提交 `{fruit_id, quantity}`、**不传价格**，服务端重算。因此**券的抵扣必须在服务端完成**，前端仅做预估展示——沿用现有"客户端预估、服务端权威"的 `verified_price`（认证价）模式。

## 二、数据模型

新增两张表，`orders` 表加 3 列。SQLAlchemy 模型写在 `backend/app/models/domain.py`，统一用 `TimestampMixin` + `Mapped/mapped_column`，风格参照现有 `CustomerAddress`。新模型需加入 `backend/app/models/__init__.py` 的 `__all__`。

### 1. `coupon_templates`（券模板 / 券种）

| 字段 | 类型 | 说明 |
|---|---|---|
| id | int PK | |
| name | String(80) | 券名，如"认证专享券" |
| description | Text 可空 | 备注 |
| discount_type | String(16) default `'amount'` | 预留扩展，本期只用 `amount` |
| amount | Numeric(10,2) | 抵扣金额 |
| min_spend | Numeric(12,2) default 0 | 满减门槛，0 = 无门槛 |
| valid_days | Integer | 领取后 N 天到期 |
| grant_on_verified | Boolean default False | 认证通过是否自动发放 |
| per_customer_limit | Integer default 1 | 每人最多持有几张（防重复发） |
| is_active | Boolean default True | 是否启用 |

### 2. `customer_coupons`（用户券实例）

发放时从模板**快照**关键字段，模板后续修改不影响已发出的券。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | int PK | |
| customer_id | FK customers.id, index | 券归属 |
| template_id | FK coupon_templates.id, index | 来源模板 |
| name / amount / min_spend | 同模板 | 发放时快照 |
| status | String(16) default `'unused'`, index | `unused` / `used` / `expired` |
| source | String(16) | `'verified'` / `'admin'` |
| issued_at | DateTime | 发放时间 |
| expires_at | DateTime, index | = issued_at + valid_days |
| used_at | DateTime 可空 | 核销时间 |
| order_id | int 可空 | 被哪个订单核销 |

### 3. `orders` 新增 3 列

- `coupon_id: int | None` —— 指向 `customer_coupons.id`
- `discount_amount: Numeric(12,2) default 0` —— 本单抵扣额
- `payable_total: Numeric(12,2) default 0` —— 实付 = `estimated_total - discount_amount`，clamp ≥ 0

> **关键设计**：`estimated_total` 的语义**保持不变**（= 商品原价合计 = Σ subtotal），新增 `payable_total` 表示实付。这样后台销售统计、订单小计的口径不受券影响，改动面最小。

## 五、发券与抵扣逻辑（新增 `backend/app/services/coupon.py`）

- `grant_coupons_on_verified(db, customer)`：取所有 `grant_on_verified=True and is_active` 的模板；对每个模板，若该客户已持有该模板的券数 < `per_customer_limit`，则新建 `customer_coupon`（快照字段，`expires_at = now + valid_days`，`source='verified'`）。**幂等**，重复调用不会重复发。
- `compute_discount(coupon, goods_total) -> Decimal`：校验门槛 `min_spend <= goods_total`，返回 `min(amount, goods_total)`。**下单核销与前端展示共用同一套口径。**

**调用点：**

1. `backend/app/api/admin.py` `review_verification`：在 `customer.verification_status = 'verified'` 之后、`db.commit()` 之前调用。
2. `backend/app/api/public.py` `_attach_phone_to_customer`：绑手机自动转 `verified` 的分支也调用——防止漏发；幂等性由 service 保证不重复。

## 六、下单核销（`backend/app/api/public.py` `_apply_order_payload`）

在设置 `order.estimated_total` 之后追加券处理。该函数被 `create_order`（POST）与 `update_order`（PATCH）复用，编辑场景自动覆盖：

1. **释放旧券**（编辑场景）：若 `order.coupon_id` 存在且 ≠ 新 `payload.coupon_id`，把旧券置回 `status='unused'`，清 `order_id` / `used_at`。
2. **应用新券**：若传了 `payload.coupon_id` —— 查该客户名下、`status='unused'`、`expires_at > now` 的券；校验 `min_spend <= estimated_total`；`discount = compute_discount(...)`；写 `order.coupon_id / discount_amount / payable_total`；券置 `used`、`used_at`、`order_id`。
3. **无券**：`discount_amount=0`、`payable_total=estimated_total`、`coupon_id=None`。
4. **校验失败**（券不存在 / 非本人 / 已用 / 过期 / 未达门槛）→ 返回 `HTTP 400`，前端提示重选。

**取消释放**（`backend/app/api/admin.py` 订单状态更新处）：订单改为 `cancelled` 时，若 `order.coupon_id` 且券未过期 → 释放回 `unused`（清 `order_id`）；已过期则保持 `expired`。

## 七、接口

### 客户端（`public.py`，要求登录）

- `GET /coupons/my?status=` —— 返回登录客户的券列表；响应时**动态判过期**（`unused` 且 `expires_at < now` 视为 `expired`）；支持 `status` 过滤。
  - 结算页取 `status=unused`，前端按门槛算是否可用；服务端下单时再权威校验。

### 管理端（`admin.py`，新增 `coupons` 权限）

- `deps.py` 权限矩阵：`super_admin` 增加 `'coupons'`。
- `GET /admin/coupon-templates`、`POST /admin/coupon-templates`、`PATCH /admin/coupon-templates/{id}`，均 `require_admin_permission('coupons')`。
- （次要，可留接口后接）`POST /admin/customers/{id}/coupons` 手动发券。

## 十、验证（端到端）

**后端**（`python -m app.db.init_db` 建表 + 跑迁移，`/docs` 或脚本驱动）：

1. 造 / 确认认证送券模板 → 后台审核某客户认证通过 → `GET /coupons/my` 应有券。
2. `POST /orders` 带 `coupon_id` → 校验 `order.discount_amount / payable_total` 正确、券转 `used`。
3. `PATCH /orders/{id}` 改单 → 抵扣重算正确、旧券释放 / 新券锁定。
4. 订单改 `cancelled` → 券释放回 `unused`。
5. 门槛不达 / 过期 / 非本人 / 重复使用 → 应返回 400。

**前端**（微信开发者工具，真机 / 模拟器）：

- 已登录且已认证账号 → 结算页出现优惠券行、自动选券、抵扣与实付正确、可切"不使用" → 提交后"我的卡券"该券转已使用、mine 页数量 -1。
- 未登录 → mine 页卡券仍 0 且不请求（回归验证既有 gate）。
- 管理端 coupons 页增改券种、开关认证送券；无 `coupons` 权限的 `order_admin` 看不到入口。
